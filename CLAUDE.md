# CLAUDE.md

このファイルは、本リポジトリでClaude Codeが開発作業を行う際の指針を記載したものである。

## プロジェクト概要

納品先売場向け「賞味期限管理Webアプリ」。現在Excelで行っている賞味期限チェック業務（バーコード読取→入力→CSV加工→Excel転記）をWebアプリに置き換え、二重入力・CSV加工作業・チェック漏れを解消することが目的。

詳細仕様は [SPEC.md](SPEC.md) を正とする。実装タスクの分解は [TASKS.md](TASKS.md) を参照。
`01_要件定義書.md` / `02_仕様設計書.md` / `03_詳細設計書.md` / `04_ER図.mermaid` は本実装に至るまでの設計ドラフトであり、参照用として残すが更新対象ではない（内容に矛盾がある場合はSPEC.mdを優先する）。

## 確定済み技術スタック

| 区分 | 技術 |
|---|---|
| バックエンド | PHP / Laravel（最新LTS） |
| 認証 | Laravel Breeze（ID/パスワード認証、Blade版） |
| フロントエンド | Laravel Blade + Tailwind CSS、JSは必要箇所のみ（バーコード読取等） |
| バーコード読取 | ブラウザカメラ + JSライブラリ（ZXing等） |
| データベース | PostgreSQL（Supabaseが提供するマネージドDBとして利用。Supabase独自のAuth/Storage機能は使用せず、接続文字列のみ利用） |
| ローカル開発環境 | Docker / Laravel Sail |
| テスト | PHPUnit（Featureテスト中心） |
| CI | GitHub Actions（PHPUnit自動実行のみ。デプロイは手動） |
| 本番ホスティング | Render（無料/低コスト枠） |

## 開発方針・規約

- ドキュメント成果物（README、設計書、コメント以外の説明文書）は日本語で作成する（ユーザーのグローバル指示）。
- マルチテナント前提のデータ設計を維持する。全テーブルに`company_id`を保持し、MVPでは`companies`を1レコード固定運用とするが、将来の企業追加はアプリ側のフィルタ条件追加のみで対応できる設計を崩さない。
- 権限モデルを徹底する：
  - 閲覧（一覧・検索）は同一企業内であれば全店舗・全担当者のデータに制限なくアクセス可能（チェック漏れの早期発見を優先するため）。
  - 登録・編集は店舗担当者は自分の担当店舗のみ、本社担当者・システム管理者は全店舗可能。
  - CSVインポート・マスタ管理は本社担当者・システム管理者のみ。
- `expiry_check_logs`（賞味期限チェック履歴）は**常に追記（INSERT）、UPDATEしない**。ロットは`(product_id, store_id, expiry_date)`の組み合わせで識別する。
- 削除は基本的に論理削除（`is_active`等のフラグ）とし、物理削除は行わない。
- 売場から商品が無くなった場合は数量0の明示登録（`is_zero_report = true`）を行う。何も登録しないことと数量0登録は意味が異なるため、この区別をコード上で曖昧にしない。
- 過去日付の賞味期限登録はバリデーションエラーとする。
- 自社マスタとJANコードの突合は「自社商品コード優先 → JANコード → 該当なしなら新規」の優先順位を厳守する（CSVインポート・バーコード読取後の商品情報取得の両方で共通のロジック）。
- Open Food Facts APIは自社マスタに該当データが無い場合のみ呼び出すフォールバックとして使う。取得結果は登録確定前まではDBに保存しない。

## コマンド（セットアップ後に整備）

```
sail up -d                  # ローカル開発環境起動
sail artisan migrate        # マイグレーション実行
sail artisan db:seed        # 開発用シードデータ投入
sail artisan test           # PHPUnitテスト実行
sail npm run dev            # Vite/Tailwindビルド（開発）
```

## 注意事項

- 性能要件：店舗数約40×品番数約100（計4,000品番規模）で一覧・検索を数秒以内に表示できること。
- 無料/低コストホスティングのリソース制約（DB容量・実行時間等）を意識した実装を行う（重いバッチ処理やN+1クエリを避ける）。
- フェーズ2（通知・電子サイン・処理履歴管理）、フェーズ3（LINE通知・マルチテナントSaaS化・権限管理・ダッシュボード）はSPEC.mdに概要のみ記載。本実装（MVP）のスコープには含めない。


## 作業の進め方

1. まず [TASKS.md](./TASKS.md)のマイルストーン順に進める
2. 各マイルストーン末尾の ** DoD(完了条件) ** を満たしたら次へ
3. SPEC.md と乖離が出そうな判断が必要なときは、勝手に決めずユーザーに確認する
4. 実装より先にドメインロジック('daysElapsed’等)のテストを書く
5. 実装が終わったらTASKS.mdのリストにある該当項目にチェックを入れてDONEにする
