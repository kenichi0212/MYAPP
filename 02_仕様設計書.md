## 3. システム全体構成

### 3.1 構成概要（論理構成）

```
[利用者（店舗担当者／本社担当者）]
        │
        ▼
[Webアプリ（フロントエンド：カメラ/バーコード読取UI、一覧・検索UI）]
        │
        ▼
[アプリケーションサーバー（PHP/Laravel）]
   ├─ 認証処理
   ├─ 商品情報取得処理（自社マスタ検索 → 未登録時のみAPI連携）
   ├─ 登録・履歴管理処理
   ├─ 検索・一覧表示処理
   └─ CSV取込処理
        │
   ┌────┴─────┐
   ▼            ▼
[自社DB]   [Open Food Facts API（外部）]
```

### 3.2 商品情報取得フロー（F03の中心仕様）

```
バーコード読取（JANコード取得）
        │
        ▼
自社マスタ（DB／CSV取込済データ）をJANコードで検索
        │
   ┌────┴────┐
 該当あり        該当なし
   │              │
   ▼              ▼
自社マスタの     Open Food Facts APIへ
商品名・メーカー名を取得         問い合わせ
   │              │
   │         ┌────┴────┐
   │       取得成功    取得失敗
   │         │          │
   │         ▼          ▼
   │   商品名・メーカー名を取得   手動入力を促す
   │         │          │
   └─────────┴──────────┘
              ▼
   登録画面に表示（取得元を画面上に表示）
              ▼
        利用者が確認・修正・削除可能
```

**業務ルール**
- 取得元が「自社マスタ」か「API」かを画面上で区別表示する（信頼度の異なるデータであるため）。
- 自社マスタ・API両方とも該当しない場合は、商品名・メーカー名を手動入力できる。
- 取得した内容はあくまで初期値であり、登録前に必ず利用者が確認できる状態とする。

## 4. 機能仕様（MVP）

### F01 ログイン機能

| 項目 | 内容 |
|---|---|
| 概要 | ID（メールアドレス等）とパスワードによる認証 |
| 入力 | ログインID、パスワード |
| 出力 | 認証成功時：ホーム画面へ遷移／認証失敗時：エラーメッセージ表示 |

### F02 バーコード読み取り

| 項目 | 内容 |
|---|---|
| 概要 | スマートフォンカメラでJANコード（バーコード）を読み取る（主たる読取方式として確定。専用リーダーは将来検討） |

| 出力 | JANコード（数値文字列） |
| 業務ルール | ・読み取り失敗時は手動でJANコードを入力できる<br>・読み取り後、自動的にF03（商品情報取得）処理へ遷移する |

### F03 商品情報取得（自社マスタ優先／APIフォールバック）

| 項目 | 内容 |
|---|---|
| 概要 | JANコードをもとに商品名・メーカー名を取得する（3.2のフロー参照） |
| 入力 | JANコード |

### F04 賞味期限・数量登録

| 項目 | 内容 |
|---|---|
| 概要 | 読み取った商品に対し、賞味期限と数量（バラ数）を入力し登録する |
| 入力 | JANコード（自動）、商品名・メーカー名（F03取得値、修正可）、賞味期限（日付）、数量（バラ数、整数） |
| 出力 | 登録完了メッセージ、登録履歴への反映 |
| 業務ルール | ・登録者（ログインユーザー）・登録日時・店舗を自動付与する<br>・商品（内部ID）＋店舗＋賞味期限の組み合わせを「ロット」とし、ロット単位で履歴を**追記**する（上書きしない。詳細は11.参照）<br>・同一商品・同一賞味期限の重複登録を検知した場合、登録時に確認ダイアログを表示し、「数量を加算する」か「別履歴として登録する」かを利用者に選択させる<br>・売場にロットが無くなった場合は、数量0として明示的に登録する（無言で消えるとチェック漏れと区別できないため。詳細は11.4参照）<br>・登録内容は履歴として保持し、削除ではなく論理削除（取消フラグ）を基本とする<br>・賞味期限に過去日付（期限切れ）は登録不可とし、入力時にエラー表示する |

| 業務ルール | ・賞味期限が近いものほど目立つ表示（色分け等）にする<br>・並び替え・絞り込みはF06〜F08と連動する |

## 7. CSVファイル仕様（自社マスタ取込：F09）

| 項目 | 内容 |
|---|---|
| ファイル形式 | CSV。文字コードは現時点で未確定のため、UTF-8／Shift-JISを自動判定して読み込む実装とする（詳細設計で実装方式を確定） |
| ヘッダー行 | あり（1行目がヘッダー） |
| 列順 | 固定順ではなく、ヘッダー名で列を判定する（出力元システムの出力順に依存しない設計とする） |
| 取込項目（列） | メーカー名、商品名、店舗グループコード、店舗コード、店舗名、担当者名、JANコード、事業所、自社商品コード |
| キー項目 | データベース上の主キーは**システムが自動発行する内部ID**とする。自社商品コード・JANコードはあくまで商品を識別するための属性とし、CSV取込時のマッチングには次の優先順位を用いる：①自社商品コード＋店舗で検索 → ②（自社商品コードが無い、または該当なしの場合）JANコード＋店舗で検索 → ③（いずれも該当なしの場合）新規データとして登録。詳細は本書末尾の補足参照 |
| 取込頻度・タイミング | 月1回。全店舗一括での取込のほか、チェーン（店舗グループ）単位で取込タイミングが異なる運用も想定する。そのため、取込は「全店舗一括」と「店舗グループ単位」のどちらでも実行できる仕様とする |
| 取込方式 | アップサート（存在すれば更新、なければ新規追加）。月次取込のため、前回取込分との差分（削除された商品の扱い等）も詳細設計で検討する |
| バリデーション | 必須項目の欠落チェック、JANコード形式チェック、重複行チェック、ヘッダー名の不一致チェック |
| エラーハンドリング | エラー行をスキップし、エラー内容（行番号・理由）を一覧表示。全体は中断しない方針 |

**【自社マスタのキー設計について：推奨理由】**

自社マスタは「メーカー名・商品名・JANコード」といった商品属性に加え、「店舗グループコード・店舗名・担当者名」という店舗側の属性も同じ行に持つ構成になっている。これは商品マスタ単体ではなく、「どの店舗でどの商品をどの担当者が管理するか」という**店舗×商品の割り当てデータ**であることを示している。

データベース上の主キーは、自社商品コード・JANコードのどちらでもなく、**システムが自動発行する内部ID**とする。自社商品コード・JANコードは「商品を見分けるための属性（手がかり）」として保持し、CSV取込時の既存データとのマッチングには以下の優先順位を用いる。

| 優先順位 | 条件 | 動作 |
|---|---|---|
| 1 | 自社商品コードがある | 自社商品コード＋店舗で既存データを検索し、一致すれば更新 |
| 2 | 自社商品コードが無い（または1で該当なし） | JANコード＋店舗で既存データを検索し、一致すれば更新 |
| 3 | いずれも該当なし | 新規データとして登録 |

この方式を採る理由は以下の通り。

- 自社商品コードは企業が自前で管理する一意性の高いコードだが、**自社商品コードという概念自体を持たない企業（将来のSaaS利用企業を含む）も想定される**ため、JANコードによるフォールバックを用意する
- JANコードのみに頼ると、量り売り・インストア発行商品など一意性が保証されない商品もあるため、自社商品コードがある場合はそれを優先する
- バーコード読取時の「自社マスタ優先→APIフォールバック」と同じ考え方であり、設計の一貫性が保てる
- 主キーを内部IDにしておくことで、企業ごとに使えるコードが異なっても同じテーブル構造のまま対応できる（将来のマルチテナント化にも有利）

## 11. 賞味期限チェックのデータ管理方針（履歴設計・ロット管理・未チェック検知）

### 11.1 基本方針

- 賞味期限チェックは**毎月実施**され、同じ商品でも月によって数量が変化する（売れる・処理される等）。
- データは**上書き更新ではなく、チェックのたびに履歴として追記**する。これにより「先月50個→今月30個」のような変化や、過去の状況を追跡できる。
- 商品は「**商品（内部ID）＋店舗＋賞味期限**」の組み合わせを1つの管理単位（以下「ロット」）として扱う。賞味期限が異なれば、同じ商品・同じ店舗でも別ロットとなる（＝別の入荷分として扱う）。商品の内部IDは7.のマッチング方式（自社商品コード優先→JANコードフォールバック）で特定したものを指す。

### 11.2 テーブル構成（論理レベル）

| 区分 | 役割 | 更新方法 |
|---|---|---|
| ①賞味期限チェック履歴 | 毎月のチェック結果を1行ずつ記録（ロット・数量・チェック日・担当者） | 常に追記のみ（上書きしない） |
| ②現在の状態（一覧表示用） | 一覧・検索画面に表示する「今の状況」 | ①から「ロットごとの最新チェック日の行」を抽出して算出する（別テーブルとして持たず、①からの導出でよい） |
| ③処理履歴 | 廃棄・値引き・返品等の処理実績を記録 | チェックとは別タイミングで発生するため、①とは別に追記 |

### 11.3 「未チェック・要確認」の検知ロジック

毎月チェックされなかったロットについて、「処理済みで本当に無いのか」「ただの見落としか」をシステム側で区別できるようにする。

**警告（要確認表示）の対象とする条件（すべて満たす場合のみ）**

1. 賞味期限までの残り期間が、設定した閾値未満（例：3か月未満）
2. 直近のチェック期間（今月）に、そのロットの①チェック履歴への記録がない
3. そのロットについて③処理履歴にも記録がない

上記3条件をすべて満たすロットのみを「要確認」として一覧に警告表示する。残り期間が閾値以上のロットは、チェックの有無にかかわらず警告対象としない（チェック負担を軽減するため）。

| 状態 | 残り期間 | 今月の①チェック | ③処理履歴 | 結果 |
|---|---|---|---|---|
| 期限に余裕あり | 閾値以上 | – | – | 対象外（何もしなくてよい） |
| 期限が近い・今月チェック済み | 閾値未満 | あり | – | 対象外（最新状態を把握済み） |
| 期限が近い・処理済み | 閾値未満 | なし | あり | 対象外（無くなった理由が明確） |
| 期限が近い・未チェック・未処理 | 閾値未満 | なし | なし | **警告対象（要確認）** |

### 11.4 店舗運用ルール：「数量0」の明示登録

ロットが売場から無くなった場合、「処理（廃棄・値引・返品）をした」のか「普通に売れてなくなった」のかを区別する必要がある。後者の場合、何も入力しないと「見落とし（チェック漏れ）」と区別できなくなる。

そのため、**売場に無くなったロットは、数量0として明示的にチェック登録する**運用とする。

- 数量0が登録されている → 「対応不要」と判断し、警告対象としない
- 何も登録されていない（0登録も処理履歴もない） → 「見落としの可能性あり」として11.3の警告対象に含める

この「入力が無いこと自体を見落としのサインとして扱う」考え方により、本当のチェック漏れを検知できるようにする。

### 11.5 関連機能への反映

- **F04（登録）**：ロット（自社商品コード＋店舗＋賞味期限）単位で履歴を追記。数量0の明示登録に対応する。
- **F05（一覧表示）**：表示項目に「要確認（未チェック警告）」フラグを追加。11.3のロジックで算出する。
- **F06（賞味期限検索）**：「要確認のみ表示」の絞り込み条件を追加できるようにする。
