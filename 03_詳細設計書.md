## 5. 主要処理の詳細ロジック

### 5.1 商品情報取得処理（バーコード読取時：自社マスタ優先→APIフォールバック）

```
INPUT: jan_code, store_id

1. product_store_assignments を store_id で絞り込み、
   紐づく products.jan_code = jan_code の行を検索
   → 該当あり：その products 行を採用（name_source = 'master'）
   → 該当なし：2へ

2. Open Food Facts API を jan_code で問い合わせ
   → 取得成功：商品名・メーカー名を画面に表示（name_source = 'api'、
     まだproductsには保存しない。登録確定時に保存する）
   → 取得失敗／該当なし：3へ

3. 商品名・メーカー名を空欄表示し、手動入力を促す（name_source = 'manual'）

OUTPUT: 商品名, メーカー名, name_source（画面に表示し、登録確定前に修正可能）
```

### 5.2 CSV取込処理（自社マスタ：F09）

```
INPUT: CSVファイル, scope（全店舗 or 店舗グループ）

1. 文字コード自動判定（UTF-8 → 判定失敗時 Shift-JIS を試行）
2. 1行目をヘッダーとして読み込み、列名と項目（メーカー名,商品名,
   店舗グループコード,店舗名,担当者名,JANコード,事業所,自社商品コード）を
   名前でマッピング（列の出現順は問わない）
3. 必須項目チェック（店舗名, JANコード or 自社商品コードのいずれか必須 等）
4. 各行について：
   a. store_groups は店舗グループコードで、stores は**店舗コード**(company_id, store_code)で、
      staff_master は担当者名で検索し、無ければ新規作成（アップサート）
   b. products を以下の優先順位でマッチング：
        i.  internal_product_code が値あり
            → (company_id, internal_product_code) で検索
        ii. 上記なし、または該当なし
            → (company_id, jan_code) で検索
        iii. いずれも該当なし → 新規 products 行を作成
   c. product_store_assignments を (product_id, store_id) で
      アップサート（存在すれば staff_master_id 等を更新、なければ新規作成）
5. エラー行は csv_import_errors に記録し、処理を継続（全体は中断しない）
6. csv_import_batches に結果（件数・文字コード等）を記録

OUTPUT: 取込結果（成功件数, エラー件数, エラー一覧）
```

### 5.3 賞味期限・数量登録処理（F04）

```
INPUT: product_id, store_id, expiry_date, quantity, is_zero_report, checked_by

1. 過去日付チェック：expiry_date < 本日 の場合 → エラーを返し登録不可
2. 重複チェック：同一 (product_id, store_id, expiry_date) で
   当月中の expiry_check_logs が既に存在するか確認
   → 存在する場合：確認ダイアログを表示し、
        ・「数量を加算する」 → 既存数量+今回数量で新規行をINSERT
        ・「別履歴として登録する」 → そのまま新規行をINSERT
   → 存在しない場合：そのまま新規行をINSERT
3. expiry_check_logs に1行追記（UPDATEは行わない）

OUTPUT: 登録完了
```

### 5.4 「現在の状態」一覧表示の導出ロジック（F05）

```
SELECT * FROM expiry_check_logs
WHERE (product_id, store_id, expiry_date, checked_at) IN (
  SELECT product_id, store_id, expiry_date, MAX(checked_at)
  FROM expiry_check_logs
  GROUP BY product_id, store_id, expiry_date
)
```
↑ ロットごとに最新のチェック行のみを抽出し、一覧表示用データとする。

### 5.5 「要確認（未チェック警告）」判定ロジック（F05・F06）

```
対象ロット = (product_id, store_id, expiry_date) の集合

各ロットについて、以下をすべて満たす場合に「要確認」フラグを立てる：

1. expiry_date - 本日 < 閾値（例：3か月）
2. 今月の期間内に、当該ロットの expiry_check_logs が存在しない
3. 当該ロットについて disposal_logs が存在しない

擬似SQL：
SELECT lot.*
FROM (5.4の現在の状態) lot
WHERE lot.expiry_date < 本日 + 閾値
  AND lot.checked_at NOT IN (今月の期間)
  AND NOT EXISTS (
    SELECT 1 FROM disposal_logs d
    WHERE d.product_id = lot.product_id
      AND d.store_id = lot.store_id
      AND d.expiry_date = lot.expiry_date
  )
```

### 5.6 通知バッチ処理（週次・フェーズ2）

```
週次バッチ（約1週間に1回実行）

対象ロット = (product_id, store_id, expiry_date) の集合のうち、
以下をすべて満たすもの：

1. expiry_date - 本日 < 通知閾値（例：1か月／3か月。利用者ごとに設定可）
2. 当該ロットについて disposal_logs に記録がない（処理済みでない）
3. 当該ロットの最新 expiry_check_logs が is_zero_report = true でない
   （売場に無いと明示登録されていない）

→ 上記をすべて満たすロットについて、対象担当者・店舗の users へ
  notifications を1行追記し、メール／アプリ内通知を送信する

※ 5.5（要確認・未チェック検知）と判定基準の一部が共通するため、
  共通の「アクティブなロット抽出」ロジックを再利用できる。
  違いは、5.5が「今月チェックされていない」ことも条件に含めるのに対し、
  本バッチは「処理・ゼロ登録されていない限り、毎週通知し続ける」点。
```

## 10. 確認・確定済み事項（記録）

- **店舗固有コード**：CSV列名「店舗コード」として確定。3.4の`store_code`を必須カラムとし、`(company_id, store_code)`を一意キーとして採用した
- **CSV取込単位の判断**：固定の運用ルールは設けず、取込実行時にその都度、取込担当者（本社担当者・システム管理者）が画面上で選択する仕様とする（5.2・6.2の設計のまま）
- **フェーズ2テーブルのER図への含有**：disposal_logs／e_signatures／notificationsもER図に含めて作成する（理由：productsやstoresと関連が深く、後から追加すると既存テーブルへの影響範囲を見落としやすいため）。図上はフェーズ2であることを注記する
