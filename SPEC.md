## 1. 背景・目的

現在、納品先店舗での賞味期限チェック業務は以下の手順で行われている。

1. 発注端末でバーコード（JANコード）を読み取る
2. 賞味期限を入力する
3. データを送信する
4. 賞味期限3か月以内の商品は数量を別途メモする
5. 店舗ごとにCSVを取得する
6. CSVをExcelで加工する
7. 4でメモした数量を6のExcelへ二重入力する

この業務フローの課題：

| No | 課題 |
|---|---|
| 1 | CSV加工作業が毎回発生し負荷が高い |
| 2 | 数量のメモ→Excel転記で二重入力が発生 |
| 3 | メモ忘れ・転記漏れが発生しやすい |
| 4 | 賞味期限間近商品の管理が散逸 |
| 5 | 店舗ごとの状況確認に時間がかかる |
| 6 | 担当者ごとの管理ができていない |

**目的**：賞味期限チェック業務をWebアプリ化し、入力・データ管理・検索・通知の一元化、二重入力・CSV加工作業の撤廃、店舗別・担当者別の状況可視化、賞味期限切れの未然防止を実現する。将来的には他社へ提供可能なSaaS基盤への発展も見据える。

## 4. システム構成

```
[利用者：店舗担当者／本社担当者／システム管理者]
        ↓
[Webアプリ（Laravel Blade + Tailwind CSS、カメラ/バーコード読取UI）]
        ↓
[Laravelアプリケーション（Render上で稼働）]
   ├─ 認証処理（Laravel Breeze）
   ├─ 商品情報取得処理（自社マスタ検索 → 未登録時のみAPI連携）
   ├─ 登録・履歴管理処理
   ├─ 検索・一覧表示処理
   └─ CSVインポート処理
        ↓
   ┌────┴────┐
   ▼         ▼
[PostgreSQL(Supabase)]  [Open Food Facts API（外部）]
```

将来的には企業単位でデータを分離するマルチテナント構成にし、食品流通業・メーカー等への他社SaaSとして提供することを視野に入れる（フェーズ3）。

### 4.1 商品情報取得フロー（F03）

```
バーコード読取（JANコード取得）
        ↓
自社マスタ（DB／CSV取込済みデータ）をJANコードで検索
   ├─ 該当あり → 自社マスタの商品名・メーカー名を取得（name_source='master'）
   └─ 該当なし → Open Food Facts APIへ問い合わせ
                  ├─ 取得成功 → 商品名・メーカー名を取得（name_source='api'）
                  └─ 取得失敗 → 手入力を促す（name_source='manual'）
        ↓
登録確認画面に表示（取得元を画面上に表示）
        ↓
利用者が確認・修正・削除可能
```

業務ルール：
- 取得元（自社マスタ／API／手入力）は画面上で区別表示する（信頼度の異なるデータであるため）。
- 自社マスタ・API両方とも該当しない場合は、商品名・メーカー名を手入力できる。
- 取得した内容はあくまで初期値であり、登録前に利用者が確認・修正できる状態とする。

## 5. 利用者・ロール

| 役割 | 想定利用者 | 主な操作 |
|---|---|---|
| 店舗担当者（store_staff） | 各店舗のチェック担当者 | バーコード読取、賞味期限・数量登録 |
| 本社担当者（hq_staff） | 自社の管理部門 | 全店舗一覧確認、担当者別一覧確認、マスタ管理（CSV取込） |
| システム管理者（admin） | 自社の情報システム部門 | ユーザー管理、マスタ管理、設定変更 |
| （将来）他社利用者 | SaaS契約企業の各役割 | 同上（企業単位で分離） |

権限の詳細は [13. 権限マトリクス](#13-権限マトリクス) を参照。**閲覧は同一企業内であれば役割を問わず全店舗・全担当者のデータにアクセス可能**（チェック漏れの早期発見を優先するため）。登録・編集は自分の担当店舗のみに制限する。

## 6. 機能仕様（MVP）

### F01 ログイン機能

| 項目 | 内容 |
|---|---|
| 概要 | ID（メールアドレス等）とパスワードによる認証 |
| 入力 | ログインID、パスワード |
| 出力 | 認証成功時：ホーム画面へ遷移／認証失敗時：エラーメッセージ表示 |
| 業務ルール | ・1ユーザーに「店舗」「担当者」の属性を紐付ける（一覧・検索の絞り込みに使用）<br>・パスワードはハッシュ化して保存<br>・ログイン状態は一定時間で自動失効（セッション管理）<br>・認証方式はID/パスワードのみとし、SSOは将来検討 |
| 実装 | Laravel Breeze（Blade版）を利用 |

### F02 バーコード読み取り

| 項目 | 内容 |
|---|---|
| 概要 | スマートフォンカメラでJANコード（バーコード）を読み取る |
| 入力 | カメラ映像 |
| 出力 | JANコード（数値文字列） |
| 業務ルール | ・読み取り失敗時は手動でJANコードを入力できる<br>・読み取り後、自動的にF03（商品情報取得）処理へ遷移する |
| 実装 | ブラウザのカメラAPI + JSバーコード読取ライブラリ（ZXing等） |

### F03 商品情報取得（自社マスタ優先／APIフォールバック）

| 項目 | 内容 |
|---|---|
| 概要 | JANコードをもとに商品名・メーカー名を取得する（4.1のフロー参照） |
| 入力 | JANコード |
| 出力 | 商品名、メーカー名、取得元（自社マスタ／API／手入力） |
| 業務ルール | ・自社マスタを先に検索する<br>・自社マスタに該当しない場合のみOpen Food Facts APIを呼び出す<br>・取得結果は修正・削除が可能 |

### F04 賞味期限・数量登録

| 項目 | 内容 |
|---|---|
| 概要 | 読み取った商品に対する賞味期限と数量（バラ数）を入力し登録する |
| 入力 | JANコード（自動）、商品名・メーカー名（F03取得値、修正可）、賞味期限（日付）、数量（バラ数、整数） |
| 出力 | 登録完了メッセージ、登録履歴への反映 |
| 業務ルール | ・登録者（ログインユーザー）・登録日時・店舗を自動付与する<br>・商品（内部ID）×店舗×賞味期限の組み合わせ「ロット」単位で履歴を**追記**する（上書きはしない。詳細は[12.データ管理方式](#12-賞味期限チェックのデータ管理方式履歴設計ロット管理未チェック検知)参照）<br>・売場から商品が無くなった場合は数量0として**明示的に登録**する（無記入で消すことはできず、チェック漏れと区別できるようにする）<br>・同一商品・同一賞味期限の重複登録を検知した場合、登録時に確認ダイアログを表示し、「数量を加算する」または「別履歴として登録する」を利用者が選択する<br>・賞味期限に過去日付は登録不可とし、入力時にエラー表示する |

### F05 商品一覧表示

| 項目 | 内容 |
|---|---|
| 概要 | 登録済み商品の一覧を表示する |
| 表示項目 | 商品名、メーカー名、JANコード、賞味期限、数量、店舗名、担当者名、登録日時、ステータス（処理済み等）、**要確認フラグ**（賞味期限が迫っているがチェックされていないロットを検出して一覧上で警告表示。判定ロジックは[12.3](#123-要確認未チェック警告の検知ロジック)参照） |
| 業務ルール | ・賞味期限が近いほど目立つ表示（色分け等）にする<br>・並び替え・絞り込みはF06・F08と連動する |

### F06 賞味期限検索

| 項目 | 内容 |
|---|---|
| 概要 | 賞味期限を基準に商品を検索・絞り込みする |
| 検索条件 | 1ヶ月未満／2ヶ月未満／3ヶ月未満／6ヶ月未満／任意期間（開始日・終了日） |
| 出力 | 条件に合致する商品一覧（F05形式） |
| 業務ルール | ・デフォルトの並び順は賞味期限が短い順<br>・検索条件は店舗別・担当者別の絞り込みと併用可能<br>・「要確認のみ表示」の絞り込み条件を追加できる |

### F07 店舗別一覧

| 項目 | 内容 |
|---|---|
| 概要 | 店舗を指定して、その店舗の商品一覧（履歴含む）を表示する |
| 入力 | 店舗（店舗グループコード／店舗名）の選択 |
| 出力 | 該当店舗の商品一覧 |
| 業務ルール | ・閲覧は店舗・担当者を問わず**同一企業内であれば全店舗のデータを閲覧可能**とする（チェック漏れの早期発見を優先し、閲覧範囲は制限しない）<br>・「店舗」を選択させる操作は、権限による制限ではなく**見やすくするための絞り込みフィルタ**として位置づける<br>・登録・編集（F04）は自分の担当店舗のみ可能とする（権限制御は登録・編集操作にのみ適用） |

### F08 担当者別一覧

| 項目 | 内容 |
|---|---|
| 概要 | 担当者を指定して、その担当者が登録した商品一覧を表示する |
| 入力 | 担当者の選択 |
| 出力 | 該当担当者の商品一覧 |
| 業務ルール | ・店舗別一覧と組み合わせたフィルタを可能とする<br>・F07と同様、閲覧は同一企業内であれば誰でも全担当者のデータを閲覧可能とする（自分以外の担当者を選んで見ることができる） |

### F09 CSVインポート（自社マスタ）

| 項目 | 内容 |
|---|---|
| 概要 | 自社マスタCSVファイルをアップロードしシステムに取り込む |
| 入力 | CSVファイル（メーカー名、商品名、店舗グループコード、店舗コード、店舗名、担当者名、JANコード、事業所、自社商品コード） |
| 出力 | 取込結果（成功件数、エラー件数、エラー内容一覧） |
| 業務ルール | ・主キーは内部ID。①自社商品コード（店舗で検索）→②JANコード（店舗で検索）→③新規登録、の優先順位でマッチングしアップサート（既存データは更新、新規データは追加）する（詳細は[10.CSVファイル仕様](#10-csvファイル仕様自社マスタ取込f09)参照）<br>・文字コード・列順・ヘッダー名の不一致時はエラー内容を明示する<br>・取込前にプレビュー表示し、確認後に確定する |

## 8. 外部インターフェース仕様

### 8.1 Open Food Facts API

| 項目 | 内容 |
|---|---|
| 用途 | 自社マスタに該当しないJANコードの商品名・メーカー名取得 |
| 呼び出しタイミング | F03において自社マスタ検索結果が0件の場合 |
| リクエスト | JANコード（バーコード番号）をキーに商品情報を問い合わせ |
| レスポンス想定項目 | 商品名、メーカー名（ブランド名）、その他属性（取得しても使用しない項目は無視） |
| エラー時の挙動 | 該当なし／タイムアウト／APIエラーの場合は手入力を促す |
| 利用制限への対応 | 無料利用のレート制限を踏まえ、同一JANコードへの連続呼び出しを抑える簡易キャッシュを設ける（自社マスタへの自動追加は行わず、取得結果のみ一時保持） |

### 10.1 取込時のマッチング優先順位

主キーはデータベース上の内部ID。自社商品コード・JANコードはあくまで商品を識別するための属性として保持し、CSV取込時の既存データとのマッチングには以下の優先順位を用いる。

| 優先順位 | 条件 | 動作 |
|---|---|---|
| 1 | 自社商品コードあり | 自社商品コード×店舗で既存データを検索し、一致すれば更新 |
| 2 | 自社商品コードなし（または1で該当なし） | JANコード×店舗で既存データを検索し、一致すれば更新 |
| 3 | どちらも該当なし | 新規データとして登録 |

### 10.2 自社マスタのキー設計について（推奨理由）

自社マスタは、メーカー名・商品名・JANコードといった商品属性に加えて、店舗グループコード・店舗名・担当者名という店舗側の属性を同じ行に持つ構成になっている。これは商品マスタ単位ではなく「どの店舗でどの商品をどの担当者が管理するか」という**店舗×商品の割り当てデータ**であることを示している。

データベース上の主キーは、自社商品コード・JANコードのどちらでもなく**システムが自動発行する内部ID**とする。自社商品コード・JANコードは商品を識別するための属性（揺れうる）として保持し、CSV取込時の既存データとのマッチングには10.1の優先順位を用いる。

この方式を採る理由：
- 自社商品コードは企業が独自に管理する一意性の高いコードだが、**自社商品コードという概念自体を持たない企業（将来のSaaS利用企業を含む）を想定する**ため、JANコードによるフォールバックを用意する
- JANコードのみに頼ると、量り売り・インストア発行商品など一意性が保証されない商品があるため、自社商品コードがある場合はそちらを優先する
- バーコード読取時の「自社マスタ優先→APIフォールバック」と同じ考え方であり、設計の一貫性を保てる
- 主キーを内部IDにすることで、企業ごとに使われるコードが異なっても同じテーブル構造のまま対応できる（将来のマルチテナント化にも有利）

## 12. 賞味期限チェックのデータ管理方式（履歴設計・ロット管理・未チェック検知）

### 12.1 基本方針

- 賞味期限チェックは**毎月実施**され、同じ商品でも月によって数量が変動する（売れた・処理された等）。
- データは**上書き更新ではなく、チェックの度に履歴として追記**する。これにより「先月50個→今月30個」のような変化や過去の状況を追跡できる。
- 商品は「商品（内部ID）×店舗×賞味期限」の組み合わせを1つの管理単位（以下「ロット」）として扱う。賞味期限が異なれば、同じ商品・同じ店舗でも別ロットとなる（別の在庫単位として扱う）。ロットの商品の内部IDは10.のマッチング方式（自社商品コード優先→JANコードフォールバック）で特定したものを指す。

### 12.2 テーブル構成（論理レベル）

| 区分 | 役割 | 更新方法 |
|---|---|---|
| ①賞味期限チェック履歴（expiry_check_logs） | 毎月のチェック結果を1行ずつ記録（ロット・数量・チェック日・担当者） | 常に追記のみ（上書きしない） |
| ②現在の状態（一覧表示用、論理ビュー） | 一覧・検索画面に表示される今の状況 | ①各ロットごとの最新チェック日の行から抽出して算出する（別テーブルとして持たず、①からの導出とする） |
| ③処理履歴（disposal_logs、フェーズ2） | 廃棄・値引き・返品等の処理実績を記録 | チェックとは別タイミングで発生するため、①とは別に追記 |

「現在の状態」の抽出ロジック：

```sql
SELECT * FROM expiry_check_logs
WHERE (product_id, store_id, expiry_date, checked_at) IN (
  SELECT product_id, store_id, expiry_date, MAX(checked_at)
  FROM expiry_check_logs
  GROUP BY product_id, store_id, expiry_date
)
```

→ ロットごとに最新のチェック行のみを抽出し、一覧表示用データとする。

### 12.3 「要確認（未チェック警告）」の検知ロジック

毎月チェックされなかったロットについて、「処理済みで本当に無いのか」「ただの見落とし」かをシステム側で区別できるようにする。

**警告（要確認表示）の対象とする条件（すべて満たす場合のみ）**

1. 賞味期限までの残り期間が、設定した閾値未満（例：3ヶ月未満）
2. 直近のチェック期間（今月）に、そのロットの①チェック履歴への記録がない
3. そのロットについて③処理履歴にも記録がない

上記3条件をすべて満たすロットのみを「要確認」として一覧に警告表示する。残り期間が閾値以上のロットは、チェックの有無にかかわらず警告対象としない（チェック負荷を軽減するため）。

| 状態 | 残り期間 | 今月の①チェック | ③処理履歴 | 結果 |
|---|---|---|---|---|
| 期間に余裕あり | 閾値以上 | - | - | 対象外（何もしなくても良い） |
| 期間が近い・今月チェック済み | 閾値未満 | あり | - | 対象外（最新状態が把握済み） |
| 期間が近い・処理済み | 閾値未満 | なし | あり | 対象外（無くなった理由が明確） |
| 期間が近い・未チェック・未処理 | 閾値未満 | なし | なし | **警告対象（要確認）** |

擬似SQL：

```sql
SELECT lot.*
FROM (12.2の「現在の状態」抽出結果) lot
WHERE lot.expiry_date < 本日 + 閾値
  AND lot.checked_at NOT IN (今月のチェック)
  AND NOT EXISTS (
    SELECT 1 FROM disposal_logs d
    WHERE d.product_id = lot.product_id
      AND d.store_id = lot.store_id
      AND d.expiry_date = lot.expiry_date
  )
```

### 12.4 店舗運用ルール：「数量0」の明示登録

ロットが売場から無くなった場合、処理（廃棄・値引き・返品）をしたのか、単純に売り切れてなくなったのかを区別する必要がある。後者の場合、何も入力しないと見落とし（チェック漏れ）と区別できなくなる。

このため、**売場に無くなったロットは、数量0として明示的にチェック登録する**運用とする。

- 数量0が登録されている → 「対応不要」と判断し、警告対象としない
- 何も登録されていない（0登録も処理履歴もない） → 「見落とし」の可能性があるとして12.3の警告対象に含める

この「明示登録」が無いこと自体を見落としのサインとして扱う考え方により、本当のチェック漏れを検知できるようにする。

### 12.5 関連機能への反映

- **F04（登録）**：ロット（自社商品コード／JANコード→店舗→賞味期限）単位で履歴を追記する。数量0の明示登録に対応する。
- **F05（一覧表示）**：表示項目に「要確認（未チェック警告）」フラグを追加する。12.3のロジックで算出する。
- **F06（賞味期限検索）**：「要確認のみ表示」の絞り込み条件を追加する。

- 想定規模：約40店舗 × 1店舗あたり約100アイテム → 合計約4,000品番

- パスワードはハッシュ化必須
- HTTPS通信必須
- 閲覧は同一企業内でオープン（全店舗・全担当者）とし、ロールベースのアクセス制御は登録・編集（担当店舗のみ）およびCSV取込・マスタ管理（本社担当者・システム管理者のみ）の操作系機能に適用する

### 14.4 可用性・保守性

- CSV取込時のフォーマット変更に対応できる設計とする（ヘッダー名ベースの列判定）
- 将来的なマルチテナント化を見据えた拡張性のある構成とする（全テーブルに`company_id`を保持）

- バーコード読取はスマートフォンカメラを主とし、専用リーダー対応は将来検討事項とする
- 通知の送信タイミング（5.6参照）は週次バッチ（約1週間に1回まとめて送信）とし、システム負荷を抑えた一般的な方式とする
- 電子サインの保存形式は「軽くしたい」という方針を踏まえ、画像（PNG等）ではなく署名の座標点列データ（ベクター形式）で保存する。画像化したものは保存せず、大幅に軽量（数KB程度）にできる
- 自社マスタCSVの区切り文字はCSVファイルの取得元システムの出力順に依存しない（固定列ではない）。詳細は本書7.のヘッダー名で各列を判定する方針と一致しており対応済み
- ログイン方式はID/パスワード認証のみとする。SSO（複数システムを1回のログインで利用できる仕組み）は現在の規模（約40店舗）では不要と判断し、将来のSaaS化・大企業導入時に再検討する
- フェーズ2（通知機能・電子サイン・処理履歴管理）、フェーズ3（LINE通知・マルチテナントSaaS化・権限管理・ダッシュボード）は本書では概要のみとし、着手時に別途詳細仕様を作成する
